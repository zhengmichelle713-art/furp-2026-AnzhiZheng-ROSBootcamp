ROS2 的核心架构是计算图（信息之间的数据流向）
计算图是是把上面举的例子拆分成一个闭环
计算图里面最重要的是 note topic 和 service 
action 和 parameter 一般是隐藏的
node 代表机器人能具备某种能力（负责一件小事的进程） -- topic 持续流动的数据（比方说某个人一直在说话） -- service 一次性的请求 -- action 长任务 -- patameter 可调旋钮（调整系统的状态）
整个过程就是先去继承一个 node 的消息 然后初始化它并收数据 然后再发一个数据 （收数据再发数据的过程叫做 echo 回声 收到什么就发出去什么 不做任何处理） 如果要对新加的节点做一些处理的话 就在这个流程中操作就可以了 
先用 Python 跑通图 后面再优化

每次打开终端都需要 source 一次环境变量
`source/opt/ros/humble/setup.bash`
告诉 Linux 系统去哪关于 ROS 的命令

`/rosout` 是报错的意思
输入命令行时 可以用 `Tab` 键自动补全
输入 `ros2 run demo_nodes_py talkers`
输出 `[INFO] [1781590980.879740816] [talker]: Publishing: "Hello World: 2"`
第二个中括号里面是时间戳 是具体的现在系统的时间 精确到纳秒
节点和节点之间的通信时有延迟的 非常小的延迟可能会被忽略 但是如果系统非常大 有上千个 topic 同时在发 这个演示就不能被忽略了 需要优化相关通信的质量

rqt_graph 是可视化计算图的一个工具

在构建机器人的时候 可以直接从 CAD 软件（一个机械制图用的软件）里导出来对机器人的描述 所以不需要去手动维护 也即是不需要手写 URDF
