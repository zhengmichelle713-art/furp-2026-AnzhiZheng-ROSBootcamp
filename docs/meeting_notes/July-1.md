`ROS2 SLAM + Nav2`

map 与 odom
1. 让 odom frame 和 map frame 合并为同一坐标系是错误的
   `map`原点是机器人初次建图的起始点位
   `odom` 原点是机器人每次启动导航、开始移动的起点，不是同一个位置
   如果节点强制融合两个坐标系后，地图数据会被直接“销毁”，RViz 可视化出现大面积黑白噪点     环境轮廓完全无法识别，是 SLAM 经典调试故障
2. TF 核心规则：同一条 TF 变换（map → odom）只能由一个节点发布，两个节点抢同一条 TF      直接漂移、崩溃；建图时由 SLAM 持有，纯定位模式由 AMCL 持有，二者不能同时启动争抢变换


SLAM 建图
1. 建图输入：机器人位姿估计（定位结果）+ 激光雷达`scan`扫描数据
2. 栅格规则：栅格网格组成地图，黑色栅格代表障碍物，白色栅格代表机器人可以通行空闲区域
3. 地图作用：同时是 SLAM 输出结果，也是导航模块的输入数据
4. 三类地图
   - `/map`静态栅格地图
     SLAM/map_server 发布，全局固定世界模型，用来定位、全局路径规划
   - global_costmap 全局代价地图
     基于完整静态地图，机器人雷达为探测到的远处障碍物也会提前规避，用于生成全局 Path
     规划器生成，在`map`基础上膨胀障碍物、标注禁行区，找从起点到终点的大路
   - local_costmap 局部代价地图
     以机器人为中心，仅覆盖雷达实施探测范围，处理动态、临时新增障碍物，用于局部轨迹实时      避障
     控制器生成，机器人身边小范围实时地图，识别突然出现的动态障碍，负责避障
5. 膨胀层 Inflaction 作用：
   机器人有实体尺寸、不能仅保证地盘中心点远离障碍物，需要给障碍物向外扩张安全区域，太高    周边栅格通行代价，避免剐蹭


里程计核心数学原理————优化问题
1. 机器人里程计本质是不断观测 + 不断修正位姿，观测一定会有噪声、误差，单纯积分运动会越     走越飘；优化就是用数学最小化所有误差，算出最贴合真实机器人位置的最优位姿
2. 两种信息：
   - 轮子 / IMU：告诉我机器人这一步移动了多少（预测，不准）
   - 激光 / 相机看环境：对比前后画面，算出移动了多少（观测，相对准一点）
3. 残差 = 两种信息之间的差值
   - 假设现在猜机器人在 A 点
   - 用轮子推算，它应该移动一段距离
   - 用激光看环境，它实际移动另一段距离
     两者的差，就是残差。
     猜的位置越真实，残差越小；位置越歪，残差越大。
   优化目标：调整机器人的位置，把所有残差全部变小


Path 全局路径 VS Trajectory 局部轨迹
1. Path（全局几何路线，Planner 输出）
   - 无时间、速度、加速度信息
   - 坐标系：全局`map`坐标系
   - 生成依据：机器人当前位姿、导航目标、全局代价地图
   - 失败场景：找不到通路、目标点卡在障碍物里
   Path 消息：`nav_msgs/Path`仅储存一串 PoseStamped 点位
2. Trajectory（局部执行轨迹，Controller 输出）
   - 包含时间、线速度、角速度、加速度，受机器人硬件运动约束
   - 坐标系：机器人`base_link/odom`局部坐标系
   - 硬件约束：eg. 两轮差分机器人无法横向平移
   - 失败场景：局部有障碍物、速度超限、跟踪偏移过大
   Trajectory 最终输出：Nav2 控制器下发`/cmd_vel`速度指令给底盘驱动


行为树 Behavior Tree 导航异常恢复机制
1. BT
   - 核心职能：全程监控导全流程，不参与路径、轨迹数学计算，只做任务调度、异常处理
   - 常见异常场景：机器人定位丢失、被人为搬动、动态障碍阻断路线、轨迹不可行
   - 标准恢复行为 Recovery：机器人原地 360° 自传，持续采集多帧激光扫描，重新完成全局匹      配找回 map 下绝对位姿
   - 调试典型现象：参数配置错误、TF 冲突、定位丢失时，机器人持续原地转圈，就是反复触发       修复行为
2. Nav2 三大核心服务（被行为树调度）
   - Planner Server 全局规划器
     接收导航目标，在 globla_costmap 上生成 Path 全局路径
   - Controller Server 局部控制器
     接收 Path，生成平滑运动轨迹，持续输出`/cmd_vel`控制小车
   - BT Navigation 行为树总调度
     串联规划、跟踪、故障恢复，对外提供`NavigateToPose`导航 Action
3. 最简导航流程
   RViz 发送导航目标点 → BT Navigator 接收；
   BT 调用 Planner，生成全局 Path；
   BT 把 Path 交给 Controller，实时生成速度指令/cmd_vel；
   底盘接收速度，Mujoco 小车移动；
   中途遇障碍：BT 触发重新规划；小车卡住：BT 触发原地旋转脱困


完整导航全闭环流程
激光 scan 传感器输入 → 轮速 / IMU 生成 odom 里程计 → SLAM/AMCL 全局校正得到 map 坐标系位姿、构建栅格地图 → 生成全局 / 局部两层代价地图 → 全局规划器生成几何 Path 路径 → 行为树调度下发路径 → 局部控制器生成带动力学 Trajectory 轨迹 → 下发`/cmd_vel`控制底盘移动 → 运动过程持续采集雷达数据，闭环更新定位与地图
